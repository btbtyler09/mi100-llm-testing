## Recommendations

- Use c=2 (interactive-optimal by TTFT/TPOT p99; per-user 48.3 tok/s) for interactive UX.  
- Cap per-request concurrency at c=2 to preserve sub-1s TTFT and low TPOT p99, as higher concurrency degrades latency disproportionately despite throughput gains.

- Enable early stopping at generation step ~5–10 before max tokens when token budget permits, to align TPOT reduction with actual output length patterns.  
- Prioritize request prioritization over strict FIFO when merging queues, but avoid preempting running requests—prefer queue weighting based on request type (interactive vs. batch).  
- For offline batch ingestion, use c=6–8 only when TTFT is irrelevant and sustained throughput is the sole goal, accepting degraded per-user latency.
